# Toxin Hotel Booking Platform (Backend)

Репозиторий содержит серверную часть (API) для платформы бронирования отелей **Toxin**. Проект разработан на базе Node.js с использованием архитектуры REST API и строгой типизации TypeScript.

🌐 Фронтенд-часть: **[Демо-стенд приложения](https://toxin-site-frontend.onrender.com)**

---

## Технологический стек

* **Runtime:** Node.js (v20+)
* **Framework:** Express 5
* **Language:** TypeScript 6
* **Database:** MongoDB & Mongoose
* **Validation:** Zod
* **Testing:** Vitest & Supertest
* **Linter & Formatter:** ESLint 10 & Prettier 3
* **API Documentation:** OpenAPI (через zod-to-openapi)

---

## Инструкции по развёртыванию

### Клонирование репозитория и установка зависимостей
```bash
git clone https://github.com
cd toxin-site-backend
npm install --legacy-peer-deps
```

### Настройка окружения
Создайте конфигурационные файлы в корневом каталоге на основе примеров:

* `.env` (для локальной разработки):
  ```env
  PSW_SALT_ROUNDS=your_salt_rounds
  NODE_ENV=development
  JWT_SECRET=your_secret_key
  ```

* `.env.test` (для тестовой среды). Количество раундов соли рекомендуется указать равным `1` для ускорения выполнения тестов:
  ```env
  PSW_SALT_ROUNDS=1
  JWT_SECRET=your_secret_key
  ```

## Архитектура и качество кода

### 1. Общий подход

Бэкенд представляет собой REST API, построенный на базе Express, TypeScript и СУБД MongoDB (Mongoose). Архитектура следует модульному принципу с четким разделением слоев (Роуты — Контроллеры — Модели — Схемы данных).

Ключевые архитектурные решения:
* **Schema-Driven подход:** Серверные схемы валидации являются единственным источником правды (SSOT) для автоматической сборки контракта OpenAPI и генерации клиентских типов.
* **Изоляция данных:** Защита от утечек внутренних структур БД во внешнюю среду за счет использования DTO-мапперов.
* **Предсказуемость типов:** Рекурсивный парсер URL-строк для нормализации и строгой типизации параметров запроса до слоя бизнес-логики.

### 2. Контроль качества и надежности API

Отказоустойчивость серверной части и сохранность данных контролируются на нескольких уровнях:
* **Статическая типизация:** TypeScript гарантирует строгую типизацию внутренних интерфейсов и моделей базы данных.
* **Линтинг и форматирование:** Идентичные фронтенду правила ESLint и Prettier обеспечивают консистентность кода во всех репозиториях платформы.
* **Валидация запросов:** Входящие данные, query-параметры и параметры пути (`params`) жестко фильтруются схемами Zod на уровне промежуточных слоев.
* **Тестирование:** Стабильность эндпоинтов проверяется интеграционными тестами «черного ящика» (Vitest + Supertest) в изолированном окружении.
* **Автоматизация CI/CD:** Автоматические пайплайны GitHub Actions проверяют код и обновляют типы контракта при каждом пуше.

### 3. Синхронизация контракта

Разработка ведется по принципу **Code-First (Schema-Driven)**.

Процесс автоматизации устроен следующим образом:
1. Схемы Zod описываются с указанием метаданных и примеров для OpenAPI (для этого библиотека расширяется методом `.openapi()`).
2. Функция `documentEndpoint` регистрирует маршруты в реестре `apiRegistry`, на основе которого генерируется файл контракта `openapi.json`.
3. Пайплайн GitHub Actions на каждый пуш компилирует TypeScript-типы на основе контракта и отправляет их в репозиторий фронтенда (обновление происходит только при наличии изменений).

### 4. Сквозная валидация данных

Безопасность и целостность данных контролируются на каждом этапе прохождения запроса:
1. **Сетевой уровень:** Входящий URL-запрос обрабатывается рекурсивным `customQueryParser`, который приводит строковые параметры к соответствующим JS типам (`number`, `boolean`, `null`), исключая аномалии типов до этапа валидации.
2. **Уровень промежуточных слоев:** Данные запроса (`body`, `query`, `params`) фильтруются схемами **Zod**. Идентификаторы `id` проверяются на валидность формата `ObjectId`. Ошибки валидации перехватываются `handleControllerError` и отдаются фронтенду в чистом и безопасном виде (без префиксов Express).
3. **Уровень БД:** Документы сохраняются через **Mongoose-модели**, которые обеспечивают контроль типов, обязательность полей и ограничения (уникальность, формат email, ISO timestamp и т.д.).

### 5. Механизм аутентификации

Аутентификация в приложении построена на базе **JWT**:
* Проверка подлинности вынесена в мидлвару `protect`, которая извлекает токен из заголовка `Authorization: Bearer <token>`.
* Мидлвара поддерживает флаг `isOptional`, что позволяет эндпоинту обрабатывать как авторизованных пользователей, так и анонимных гостей.
* При успешной проверке данные пользователя извлекаются из MongoDB (без пароля) и подмешиваются в объект запроса (`req.user`).
* Авторизация реализуется через мидлвару `authorize(role)`, которая проверяет роль текущего пользователя.
